# APS-300 — Evidence Model

Document ID: APS-300  
Version: 1.0-DRAFT  
Status: DRAFT  
Classification: Normative Specification  
Authority: APS-001 · APS-100 · APS-200  
Last Review: 2026-07-23

---

## 1. Purpose

APS-300 defines the canonical Evidence model used by the Aura Protocol.

Every execution of a conformant implementation MUST be capable of generating Evidence conformant with this specification.

---

## 2. Objectives

Evidence MUST provide:
- **Auditability** — an auditor can independently verify what happened
- **Integrity** — any tampering is detectable
- **Traceability** — evidence links to the requirement it documents
- **Replayability** — the execution can be reproduced with identical results
- **Independent Verifiability** — no access to the original system is required

---

## 3. Evidence Principles

Every Evidence artifact MUST be:
- **Immutable** — MUST NOT be modified after generation (INV-004)
- **Deterministic** — generated by a deterministic process (INV-001)
- **Traceable** — links to the APS requirement it documents (INV-005)
- **Verifiable** — cryptographic integrity is checkable (INV-011)
- **Versioned** — carries protocol and schema versions (INV-009)

---

## 4. Evidence Types

| ID | Type | Description |
|----|------|-------------|
| EVID-CORE | Core Evidence | Proof of a single execution |
| EVID-AUDIT | Audit Evidence | Extended audit trail evidence |
| EVID-CONF | Conformance Evidence | Evidence of passing conformance tests |
| EVID-REL | Release Evidence | Evidence associated with a release |
| EVID-CHAIN | Chain Evidence | Links to a previous Evidence entry |

---

## 5. Canonical Evidence Object

Every Evidence object MUST contain at minimum:

| Field | Type | Requirement | Description |
|-------|------|-------------|-------------|
| `evidence_id` | string | MUST | Globally unique identifier (UUID v4) |
| `protocol_version` | string | MUST | APS version (e.g., `1.0`) |
| `schema_version` | string | MUST | APS-300 schema version |
| `implementation_id` | string | MUST | Identifier of the implementation (ENT-008) |
| `execution_id` | string | MUST | References the Protocol Header execution_id (ENT-001) |
| `timestamp` | string (ISO 8601) | MUST | Time of evidence generation (UTC) |
| `policy_reference` | string | MUST | object_id of the Policy Reference (ENT-004) |
| `input_hash` | string | MUST | SHA-256 hash of the canonical input |
| `output_hash` | string | MUST | SHA-256 hash of the canonical output |
| `evidence_hash` | string | MUST | SHA-256 hash of this Evidence object (excluding this field) |
| `previous_evidence_hash` | string | SHOULD | Hash of the previous Evidence object (if chain exists) |
| `attestation_reference` | string | MUST | object_id of the associated Attestation (ENT-006) |

## 6. Evidence Pack

An Evidence Pack is the complete set of artifacts describing a single execution.

Minimum contents:

| Component | Type | Requirement |
|-----------|------|-------------|
| Evidence Object | APS-300 §5 | MUST |
| Evaluation Result | ENT-003 | MUST |
| Policy Reference | ENT-004 | MUST |
| Attestation | ENT-006 | MUST |
| Integrity Metadata | object | MUST |

## 7. Integrity Requirements

Evidence MUST:
- Enable detection of any modification (INV-004)
- Maintain cryptographic integrity (INV-011)
- Have a unique identifier (INV-015)
- Be linked to the corresponding APS version (INV-009)

---

## 8. Evidence Chain

If an implementation stores execution history, successive Evidence objects MAY form a chain via `previous_evidence_hash`.

The specification does not mandate a specific cryptographic algorithm but requires that chain integrity be independently verifiable.

Evidence chain type: `EVID-CHAIN`

---

## 9. Verification

Every Evidence object MUST enable verification of:
- APS conformance of the attested execution
- Protocol Invariant compliance
- Data integrity (each hash matches)
- Completeness of required fields
- Correctness of Evidence Pack linkage

---

## 10. Retention & Archival

The specification defines structure and verifiability requirements. Retention periods and archival policies are determined by the implementation or applicable regulations.

---

## 11. Traceability

```
APS Requirement
        ↓
Protocol Invariant
        ↓
Evaluation Execution
        ↓
Evidence Object
        ↓
Conformance Test
        ↓
Release
```

---

## 12. Failure Conditions

Evidence is invalid if:
- Required fields are missing
- Integrity has been violated
- The link to the execution cannot be verified
- APS-300 requirements are not met
